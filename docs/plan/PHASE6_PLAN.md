# Phase 6 실행계획 — 히스토리 / 추이 / 비교

> **상위 문서**: PLAN.md §7(핵심 기능 5), §6(GET /api/analyses), Phase 3 버전/MDC95 정합
> **기간(현실안)**: 1–2주
> **완료 기준**: 기간별 변화 추이 그래프
> **성격**: 누적된 분석을 시계열로 보여주고, 두 분석 간 변화를 판단(MDC95)해 의미 있게 비교.

---

## 0. 목표 한 줄

사용자의 과거 분석을 목록·기간별 추이 그래프로 보여주고, 두 시점 결과를 비교해
**변화가 통계적으로 의미 있는지(MDC95)** 함께 제시한다.

---

## 1. 선행 조건

- [ ] Phase 5 완료(단건 결과 시각화 컴포넌트 재사용 가능)
- [ ] 여러 건의 `done` 분석 데이터 확보(테스트용 시드 포함)
- [ ] `engine_version`/`score_schema_version` 저장 확인(Phase 3) — 비교 가능 여부 판단 근거

## 2. 작업 단위

### 2.0 의존성 설치
```bash
# 차트 라이브러리는 PHASE5에서 설치됨 (recharts)
# 추가 패키지 필요 시 여기에 명시
```

### 2.1 히스토리 목록
- [ ] `GET /api/v1/analyses` 페이지네이션 연동(TanStack Query — Day 19-21 패턴)
- [ ] 목록 카드: 날짜·`overall_score`·`skin_type`·썸네일(서명 URL)·상태
- [ ] 정렬(최신순) — 인덱스 `(user_id, created_at DESC)` 활용
- [ ] 빈 상태/로딩/에러 UI

### 2.2 기간별 추이 그래프 (완료 기준)
- [ ] 백엔드: 기간 파라미터로 시계열 점수 반환(또는 목록에서 클라이언트 가공)
- [ ] `overall_score` 및 주요 카테고리의 시간축 라인 차트
- [ ] 기간 필터(최근 1/3/6개월 등), 카테고리 토글
  ```bash
  # .env.local
  DEFAULT_PERIOD_MONTHS=3
  ```
- [ ] 모바일 반응형

### 2.3 두 분석 비교 (MDC95)
- [ ] 비교 대상 2건 선택 UI
- [ ] 항목별 변화량 + **변화 판정 라벨**: `STABLE / UNCERTAIN / IMPROVED / WORSENED`
  > SkinLens `reliability.py`의 MDC95 기반 변화 판정 결과를 활용
- [ ] **버전 정합 가드**: `score_schema_version`이 다르면 직접 점수 비교 경고/제한
      (스키마가 바뀐 두 분석을 단순 비교하면 오해 소지)
- [ ] before/after 히트맵·이미지 나란히 보기(과거 Excel/HTML 비교 리포트의 웹판)

### 2.4 백엔드 보완(필요 시)
- [ ] 추이/비교용 조회 엔드포인트 또는 쿼리 최적화
- [ ] 변화 판정을 어디서 계산할지 결정: 워커 저장 시 vs 조회 시 (재현성 위해 저장 권장)

### 2.5 PWA 히스토리 캐싱 (오프라인 지원 확장)
- [ ] 히스토리 목록 오프라인 캐싱 (IndexedDB 또는 Service Worker Cache)
- [ ] 추이 그래프 데이터 오프라인 캐싱
- [ ] 비교 결과 오프라인에서 확인 가능
- [ ] 오프라인 상태에서 캐시된 히스토리/추이 조회 가능
- [ ] 캐시 만료 정책 (최신 데이터 유지)

## 5. 리스크 / 주의

- **버전 혼합 비교 함정**: `score_schema_version` 차이를 무시하면 추이가 거짓 신호를 낸다 → 가드 필수.
- **MDC95 정의 일관성**: 변화 판정 기준은 SkinLens 본체와 동일 정의를 따른다(이중 정의 금지).
- **N+1/성능**: 목록 썸네일 서명 URL 대량 생성 시 비용·지연. 배치/캐시 고려.
- **표현 규제**: 추이를 "호전/악화"로 적되 "진단·치료 효과" 단정 금지(§8).
- **캐시 용량**: 히스토리 캐싱 시 PWA 저장소 용량 제한 고려
